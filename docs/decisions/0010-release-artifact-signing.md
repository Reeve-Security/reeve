# ADR-0010: Release artifacts are signed as keyless cosign bundles

- **Status:** Accepted 2026-04-26
- **Decides:** Release artifact signing format and workflow for GitHub issue #27
- **Related:** ADR-0004, ADR-0006, ADR-0007, `README.md`, `.github/workflows/release.yml`

## Context

ADR-0004 defines how Reeve signs scan output. ADR-0006 says real
signing uses `cosign` and must fail closed rather than falling back
to fixture output. ADR-0007 proves the live scan-output signing path
in a dedicated workflow.

GitHub issue #27 identifies a separate release-integrity gap: the
`cargo-dist` release workflow builds and publishes binary artifacts,
but those release artifacts were not themselves signed. That leaves
users with source and binary hashes from GitHub release assets but no
keyless Sigstore identity binding for the downloadable archives.

The release artifact signer must satisfy four constraints:

- no long-lived signing keys,
- a public GitHub Actions OIDC identity that users can verify,
- signatures published next to the release assets, and
- clear verification commands in the install docs.

## Options considered

### A. Detached `.sig` and `.crt` files from `cosign sign-blob`

Run `cosign sign-blob` for each artifact and publish a detached
signature plus the signing certificate.

Pros: familiar to older cosign users; output files are small.

Cons: verification requires keeping two sidecar files together and
does not carry the transparency-log material as one object. Users
must understand which certificate and signature belong to which
artifact.

### B. Sigstore bundle sidecars from `cosign sign-blob` *(chosen)*

Run `cosign sign-blob --yes --bundle "${artifact}.bundle"
"${artifact}"` for each selected release artifact. The bundle
contains the signature, Fulcio certificate material, and Rekor
transparency-log proof needed by `cosign verify-blob`.

Pros: one sidecar file per artifact; verification command is short;
matches ADR-0004's preference for bundle-shaped Sigstore evidence;
works with GitHub Actions OIDC and no long-lived keys.

Cons: bundle files are larger than raw signatures, and the release
workflow must verify the bundle before upload to catch signer
misconfiguration.

### C. Rely on GitHub-generated provenance only

Use GitHub release asset checksums and defer artifact signing to a
later SLSA provenance workflow.

Pros: minimal implementation work.

Cons: fails the issue requirement to sign the binary tarballs,
source tarball, and policy bundle now. It also gives users no
project-controlled cosign verification recipe for v0.1.

## Decision

Reeve signs release artifacts in `.github/workflows/release.yml`
after `cargo-dist` has produced and staged artifacts and before the
GitHub Release is created. The workflow stages a source tarball with
`git archive` and copies the committed
`crates/aibom-policy/bundles/0.1.0.wasm` policy bundle into the
release artifact directory when it exists. The publishing job
requests `id-token: write` and uses GitHub Actions OIDC keyless
signing. It installs pinned `cosign v3.0.6` using the same
SHA-256-checked binary source as the live Sigstore acceptance
workflow.

The exact release signing invocation is:

```bash
cosign sign-blob --yes --bundle "${artifact}.bundle" "${artifact}"
```

The workflow signs each staged release archive matching
`*.tar.gz`, `*.tgz`, `*.tar.xz`, `*.zip`, or `*.sh`, and signs the pre-built
policy bundle `0.1.0.wasm` when that file is present in the staged
release artifacts. This covers the cargo-dist binary tarballs, the
workflow-staged release source tarball, the cargo-dist shell installer, and the policy WASM bundle
without signing derived checksum files.

Before publishing, the workflow verifies every generated bundle with:

```bash
cosign verify-blob \
  --bundle "${artifact}.bundle" \
  --certificate-identity-regexp "^https://github.com/${GITHUB_REPOSITORY}/.*" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  "${artifact}"
```

The release uploads the artifact and its `.bundle` sidecar together.
The README install section documents the same verification path for
users.

Because `.github/workflows/release.yml` is generated by `cargo-dist`
and then deliberately extended with Reeve-specific signing steps, the
workflow invokes `dist` with `--allow-dirty`. The static contract check
in CI verifies those local signing extensions so `cargo-dist` generation
drift does not silently remove or weaken the release-signing contract.

## Rationale

Bundles preserve the security property Reeve already uses for scan
output: the user receives one verification object that carries the
signature material and transparency evidence together. That reduces
operator error compared with separate `.sig` and `.crt` sidecars.

Signing in the existing `host` job keeps the release flow simple:
`cargo-dist` builds and stages artifacts, the signing step adds
bundle sidecars to the same `artifacts/` directory, and `gh release
create` uploads the complete set. The signing step runs only on
publishing release runs because the `host` job already skips pull
requests.

Using GitHub Actions OIDC honors ADR-0006's no-long-lived-key rule.
The workflow must request `id-token: write`; without that
permission, cosign keyless signing cannot obtain the short-lived
identity token and the release fails closed.

## Plain-language summary

The release archive is what users actually install, so it needs its
own signature. Signing the AIBOM output proves an inventory file came
from a trusted signer; signing the release archive proves the binary
download came from Reeve's release workflow.

We chose the Sigstore bundle format because it packages the proof in
one file. For an archive named `aibom-cli-x86_64-unknown-linux-gnu.tar.xz`,
the release also publishes
`aibom-cli-x86_64-unknown-linux-gnu.tar.xz.bundle`. Users download
both files and run `cosign verify-blob`.

No maintainer key is stored in GitHub. During the release, GitHub
Actions asks Sigstore for a short-lived certificate tied to the
release workflow identity. Anyone can later verify that identity and
the Rekor transparency entry with cosign.

This is separate from Reeve's scan-output signing. The same tool
(`cosign`) and the same no-long-lived-key model are used, but the
subject being signed is the downloadable release asset.

## Consequences

- **This decision commits the project to:** publishing `.bundle`
  sidecars for signed release archives and installer scripts; using GitHub Actions OIDC
  instead of long-lived signing keys; pinning the release workflow to
  cosign v3.0.6 until an ADR or follow-up issue changes the version.
- **This decision unblocks:** GitHub issue #27 and v0.1 release
  distribution with verifiable binary artifacts.
- **This decision forecloses:** unsigned binary release tarballs and
  release signing with repository secrets or maintainer-held private
  keys.
- **This decision defers:** SLSA provenance generation and native
  Rust release signing without the cosign CLI.
