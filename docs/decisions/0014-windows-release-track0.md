# ADR-0014: Track 0 adds a Windows release-build target without Windows discovery or profiling

- **Status:** Accepted 2026-04-28
- **Decides:** Issue #57 Track 0 release-runner/toolchain viability slice
- **Related:** ADR-0010 (release artifact signing), ADR-0012 (CI runner substrate), `dist-workspace.toml`, `.github/workflows/release.yml`

## Context

Issue #57 asks Reeve to start validating the Windows release path without expanding the product surface into Windows discovery, sandboxing, ETW, or AppContainer enforcement. The first slice is intentionally narrow: prove that the CI substrate, Rust toolchain, and cargo-dist configuration can plan and build an `x86_64-pc-windows-msvc` artifact.

The current cargo-dist workflow already computes its build matrix from `dist-workspace.toml` via `dist plan`. Pull requests run the release workflow in plan mode, while tagged pushes publish. Track 0 therefore adds a separate PR/workflow-dispatch probe that runs only the Windows cargo-dist local-artifact build on Blacksmith Windows; this exercises the runner and MSVC toolchain without widening the full release workflow or tagging a release.

## Findings captured for Track 0

- Blacksmith's Windows x64 Public Beta runner labels include `blacksmith-2vcpu-windows-2025`, `blacksmith-4vcpu-windows-2025`, `blacksmith-8vcpu-windows-2025`, `blacksmith-16vcpu-windows-2025`, and `blacksmith-32vcpu-windows-2025`.
- Blacksmith Windows runners use the official GitHub Windows runner image with documented exclusions; Visual Studio Build Tools 2022 are available for the `x86_64-pc-windows-msvc` Rust target.
- The release workflow installs the current stable Rust toolchain for native matrix jobs before `dist build`; the Windows runner image's preinstalled Rust can lag behind Reeve dependencies' MSRV.
- Docker Linux containers are not supported on Blacksmith Windows runners. Reeve's cargo-dist local-artifact job must remain a native Windows job for Windows artifacts.
- `sigstore/cosign-installer` metadata supports `runner_os == 'Windows'` and installs the `cosign-windows-amd64.exe` asset. Reeve's release signing job still signs from the Linux host job today, but CI can install cosign on Windows if a future Windows-specific acceptance step needs it.

## Decision

Add `x86_64-pc-windows-msvc` to the cargo-dist target list as a release-build-only target. The target validates that cargo-dist can plan a Windows local-artifact job on Blacksmith Windows and that Rust/MSVC build tooling is available there. It does **not** add Windows discovery, profiling, sandbox enforcement, ETW collection, or Windows path grammar to the AIBOM schema.

The expected cargo-dist runner for the Windows local-artifact matrix is `blacksmith-4vcpu-windows-2025`. The release workflow remains generated by cargo-dist and keeps using `runs-on: ${{ matrix.runner }}` so the runner label is derived from the dist plan rather than duplicated by hand in release YAML. The dedicated Track 0 workflow pins the same runner label and executes `dist build --artifacts=local --target=x86_64-pc-windows-msvc` plus a cosign installer smoke check on Windows.

## Rationale

This keeps the slice small and reversible. The highest-risk unknown for a future Windows release is whether managed Windows runners, MSVC tooling, and cargo-dist's generated GitHub Actions matrix agree. CI can answer that question before Reeve implements Windows-specific discovery or profiling behavior.

The scope boundary remains unchanged: Windows sandbox support is still deferred. A Windows binary is a distribution artifact; it is not a claim that Reeve can observe Windows-specific capabilities yet.

## Plain-language summary

This change teaches the release machinery to try building the Windows executable. It does not teach Reeve how to scan Windows differently.

The important question for this slice is operational: when cargo-dist sees a Windows target, can it send the job to a Blacksmith Windows machine that has the Microsoft compiler tools Rust needs? If that works in CI, later Windows product work starts from a known-good build lane instead of debugging the runner and the feature at the same time.

Cosign support was checked as part of the same release path. The current signing job runs on Linux after artifacts are built, but the cosign installer action has a Windows path and a Windows AMD64 asset if Reeve later needs a Windows job to perform a signing-related acceptance check.

## Consequences

- **This decision commits the project to:** keeping `x86_64-pc-windows-msvc` in `dist-workspace.toml`, preserving cargo-dist's generated matrix-driven release workflow, documenting the expected `blacksmith-4vcpu-windows-2025` runner path, and keeping the dedicated Track 0 Windows build probe green.
- **This decision unblocks:** CI verification of Blacksmith Windows runner availability, MSVC toolchain availability, and cargo-dist Windows artifact planning/building.
- **This decision forecloses:** treating Windows release artifacts as evidence of Windows discovery/profiling support.
- **This decision defers:** Windows discovery paths, Windows capability profiling, ETW, AppContainer or other Windows sandbox enforcement, Docker-on-Windows release jobs, and any Windows schema/path grammar changes.
